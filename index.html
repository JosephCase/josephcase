A fully responsive artist portfolio website with a custom made CMS.

A project developed in my spare time, it consists of a fully responsive portfolio website written in PHP (www.giusypirrotta.com) and a Node.js CMS. The CMS enables to user to create and edit pages, text, image and video content, by writing to the mySQL database and saving associated files. The website then reads from the database, serving the content directly or using it to reference a saved file.

Key feature - Image optimisation

It was important that the website could display high quality images at multiple sizes and across multiple devices, while still achieving fast load times. In order to achieve this the CMS converts uploaded images to various file sizes using Imagemagick cli. The website itself is then chooses the most appropriate image size to serve, based on the physical display width of the image on the page.

The website

The website uses the Mod_rewrite engine to route all requests to a single PHP script, which serves the relevant content from the mySQL database based on the HTTP request. The CMS inputted text is formatted using Markdown and the website has a simple session language toggle, which controls which content is displayed based on its language setting in the CMS.

The CMS

The CMS uses the Node Express framework to route HTTP requests, serve static content and handle user session. The user is able to create new and edit existing pages, and within a page create, update and delete image, video and text content.

When editing a page the user it given graphical representation of the changes due to take place and can then save those changes once they are happy. When a page is saved the multiple changes are submitted to the server as a JSON set of instruction objects, with and associated files uploaded using FormData object. On the server the instructions are then processed asynchronously, updating the database and creating, updating or deleting any relevant files.

A responsive travel website with pre-compiled CMS data for faster load times.

During my time at Intuitive (www.intuitivesystems.com) I took the lead in the development of a new client website for Classic Collection (www.classic-collection.co.uk). The website reads for a SQL "Big C" data table which stores pre-processed CMS information as XML for fast read times. The website CMS offers the client a high level of customisability including the ability to override the natural data hierarchy.

Application Structure

The website structure is based on an XML map which defines page types and the 'widgets' that should appear on them. A 'widget' in this context is VB.Net class which performs some function on the page, usually displaying a section of content such as an image gallery or a Google maps feature.

Generally speaking each widget will correspond to a row within the "Big C" data table, if it is page specific identified by the widget type, page type and page id. The "Big C" table contains pre-compiled XML representing the CMS data for the unique instance of a widget on a page. The widget class will pull the data from the "Big C" table and initially present it on page using XSLT. For widgets where not all the data is displayed immediately or were greater level of functionality is required (for example a results page with filters and pagination) the "Big C" XML is parsed and stored within the instance of the widget class, so that it can be manipulated and presented in response to later AJAX calls.

CMS Control

The CMS is built within the companies iVector CMS structure, where each page or section is represented by a CMS page and a corresponding CMS table. These can be bespoke CMS pages, or pages based on the existing data in the iVector core database, for example a CMS page which allows the client to customise Hotel pages. At a set interval a stored SQL script will pull the required information for each widget from CMS and core tables and write it as XML to the "Big C" data table.

The CMS also gives the client the ability to override the data from the iVector core tables and change the natural hierarchy. The central content for the website is based on a Country > Region > Resort > Hotel hierarchy and the CMS gives the client the ability to show and hide Countries, Region, Resorts and Hotels, but also present Countries, Regions and Resorts interchangeably, for example changing a particular country (somewhere like Monaco) to a hierarchy of Country -> Hotel, so that while it will appear as a country, it will behave as a Resort (i.e. the parent of Hotels).

Other website features

  • Cross device responsive design.
  • Autocomplete search tool able to search Countries, Regions, Resorts and Hotels.
  • Paginated hotel results pages with filters.
  • Google maps with markers and custom info windows.
  • Responsive image carousels.
  • Agent and customer login areas, with ability to view and edit bookings.
  • User forms (e.g. Request a Quote page).
  • Recently viewed hotels cache.
  • Page redirects.

A development tool that converts layered psds into fully responsive cross device ad units.

Developed during my time at Bluequest (www.bluequest.co.uk), this ad building tool developed in PHP allows our developers to create and edit the layout of cross device, fully responsive advertising unit, by simply dragging and dropping psds or images onto a work area - greatly improving our productivity.

Once the ad has been created the user can edit its layout using the AdBuilder's editing tools and then has the freedom to develop bespoke features, such as navigation, animations and canvas features using custom code.

Core concept

The ad is represented by a hierarchical PHP class structure Ad > View > Element. The application processes each layer of an uploaded psd, saving the image and storing it's size, position and reference to its image as an Element object. The Ad object then writes a HTML and CSS file representing the PSD in HTML form and saves the Ad object as JSON, so that it can be recalled and edited later.

Cross-device support

The AdBuilder supports multiple device designs within a single unit through use of the View class. PSDs can be uploaded for each 'view' (e.g. Desktop, Tablet and Mobile) and our independent Scale_Controller.js module controls which view to show and the corresponding images to load based on the browser window size.

Editing

The AdBuilder features a set of editing tools which enable the user to add content and edit the layout of an ad that it has created. It does this by re-instantiating the Ad object from it's saved JSON and then updating it based on the changes made, before rewriting the HTML, CSS and re-saving itself as JSON.

The editing tools allows the user to;

  • Create new Elements by uploading images or psds.
  • Change Element anchor positions.
  • Control Element scale behaviour.
  • Group/ungroup Elements (the PHP Element objects can be nested).
  • Delete Elements.
